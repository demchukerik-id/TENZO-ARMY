## Day 1 (2026-04-23, Wed): triage and discovery

### F5 stuttering drain
- User saw "13 entries logged, should be 59" mid-day in F5
  shipping log. Drain rate was ~0.3–0.8 records/min.
- Found 5 root causes (full breakdown in
  `solved/F5/2026-04-23-f5-stuttering-drain.md`):
  1. `CIRCUIT_BREAKER_THRESHOLD = 2` too tight
  2. `markQueuePoll` never resets `rate_limit_hits` on success
  3. Location-fetch 429 misclassified as "location unresolved"
  4. No global Katana request budget across flows
  5. Self-kick races with cron
- Fix #1–#3 deployed Day 2 as v23.
- Manual recovery during the incident: 3× soft-reset of
  `f5_check_queue` (`UPDATE … SET backoff_until=NULL,
  rate_limit_hits=0`) + manual kick. Drained 84/84 by EOD.

### MO-7813 duplicate WK rows (3 entries instead of 1)
- Three identical WK rows in F4 activity log within 30 ms.
- Root cause: `acquire_mo_poll_lock(p_locked_by text DEFAULT
  'mo_sync')` — every concurrent caller sees the same default
  identity, so all "acquire" the same lock and proceed to insert.
- Fix Day 2: SQL migration to make default unique
  (`gen_random_uuid()`), partial unique index on
  `mo_sync_records(mo_id) WHERE status IN ('recorded','processing')`,
  and explicit caller ID in `mo-sync/index.ts` with 23505
  duplicate handling.

### PO-648 lot race (Reverted → Received cycles)
- `reconcile_mixed` records doing revert+failed-add as a pair
  every reconcile tick. WASP would briefly drop 840 EGG-X then
  recover on the next sync. Brief but real inventory dip.
- Root cause: `po-sync` snapshotted the PO before Katana finished
  re-attaching the receipt's batch_transactions. `src_lot=null`
  → WASP rejected the add → revert ran successfully → next pass
  picked up the lot → re-added.
- Fix Day 2: lot-pending guard in `po-sync` — defer the line
  (skip both add and revert) when `src_lot` is null on a row
  whose Katana data has `batch_transactions`. Record stays
  "skipped" with `lot_pending` reason.
- Doc explicitly flagged that the **looping itself** wasn't fixed
  by this — only the inventory damage. Day 2 brought the actual
  loop fix (hysteresis).

### ST flow webhook silence (9 days)
- ST-393 was flagged by Robyn — failed because Katana asked for
  56 units but WASP only had 52. Operator entry error, not a
  bug.
- But discovered: zero `stock_transfer.*` webhooks since 2026-04-14.
  ST flow has been kept alive only by the reconcile cron.
- Plus: `partial` ST sync_records never get retried — same
  `findActiveRecord`-style dedupe that makes any post-failure
  fix invisible to reconcile.
- Both documented in `active/ST/2026-04-23-st-no-retry-and-webhook-silence.md`.

### Recurring revert+receive on PO-648 after lot-pending fix
- Even after the lot-pending guard, PO-648 kept cycling: every ~1
  hour a `reconcile_revert` followed by `reconcile.received`.
- Root cause hypothesis: Katana returns `quantity_in_base_unit`
  with sub-unit float drift between calls. Our `0.0001` tolerance
  in three comparison sites was too tight.
- Fix moved to Day 2.

## Day 2 (2026-04-24, Thu/Fri): fixes deployed

### F5 v23 — drain-speed fixes
**Deployed:** Edge Function `f5-sync` v24 on `dxrkhcxiikynqybjgwys`.

- Raised `CIRCUIT_BREAKER_THRESHOLD` from 2 → 5
- `markQueuePoll` resets `rate_limit_hits = 0` on successful poll
- `resolveLocationName` returns `null` (instead of `""`) on
  transient failure; `snapshotSO` propagates as
  `error: "location_fetch_transient"` so caller takes the right
  backoff path
- Doc moved from `active/` to `solved/` with full Applied Fix
  section.

### F5 v25 — drain tightening
**Deployed:** Edge Function `f5-sync` v25.

- `INTER_ITEM_DELAY_MS`: 300 → 150
- `BACKOFF_LADDER_MS[0]`: 30s → 15s
- Soft-reset of `f5_check_queue` to clear legacy `hits>=2` state.

Hit a real ceiling: even with these fixes, drain is ~1/min when
the queue has work because **Katana's API rate limit** is the
binding constraint. Each F5 SO costs ~5–10 Katana calls; aggregate
QPS across all flows triggers 429s. New active doc captures this:
`active/F5/2026-04-24-throughput-ceiling.md`.

### MO duplicate-syncs full fix
**Applied via:** Supabase MCP `apply_migration` (SQL) + CLI deploy
(`mo-sync/index.ts`).

- Patched `acquire_mo_poll_lock` SQL function to generate unique
  caller ID when `p_locked_by` is NULL.
- Added `CREATE UNIQUE INDEX CONCURRENTLY mo_sync_records_one_active
  ON mo_sync_records(mo_id) WHERE status IN ('recorded','processing')`
  as DB-level safety net.
- Updated `mo-sync.acquireLock` to pass unique caller ID, and
  `saveRecord` to return `"duplicate"` on 23505 unique-violation
  (handled cleanly in `processOne`).

### MO empty-snapshot defer guard
- MO-7827 reverted then re-marked Done; mo-sync committed sync_id
  166 with `status=done` and **zero lines** because Katana race
  returned an empty `movements` array.
- Fix: `mo-sync` refuses to commit `done` with 0 lines — defers
  instead, queue stays for next reconcile.
- Doc: `solved/MO/2026-04-23-mo-rebuild-empty-snapshot.md`.

### PO v8 — hysteresis on delta comparisons
**Deployed:** Edge Function `po-sync` v8.

- `COMPARE_TOLERANCE = 0.5` replaces `0.0001` at all three
  delta-comparison sites in `processOne`.
- Real receipts are always whole-unit; 0.5 is comfortably below
  any legitimate delta and well above sub-unit float noise.
- Verified: PO-648 stopped cycling immediately. Doc:
  `active/PO/2026-04-24-po-reconcile-hysteresis.md`.

### PO v9 — stuck-after-partial guard
**Deployed:** Edge Function `po-sync` v9 / version 11 in Supabase.

- New guard before `saveRecord`: if the most recent sync_record
  for this PO is `partial` AND the planned line set is byte-
  identical to the previously-failed line set, skip and require
  operator Resync.
- Stops PO-726 (TEST-SUP-SHIPBOX with null lot) from spamming
  WASP with the same 422 error every minute.
- Verified: zero new PO-726 sync_records since deploy.
- Doc: `active/PO/2026-04-24-po-stuck-after-partial.md`.

### SA WASP-TEST-001 lot mismatch
- User created WASP-TEST-001 in WASP with stock=1 lot=LOT-TEST-001.
- WASP fired SA webhook → fallback resolved the lot (despite
  `{trans.Lot}` placeholder coming through unsubstituted) → Katana
  refused because `LOT-TEST-001` doesn't exist in Katana.
- This is **not a code bug** — it's a WASP↔Katana item-config
  drift problem with no recovery path once stock>0 (because WASP
  locks edits on items with stock).
- Doc: `active/SA/2026-04-24-wasp-katana-item-config-drift.md`,
  with 4 mitigations ranked (M1 pre-sync validation gate is the
  most impactful).

### MO-7854 "Low Stock" on IP-EGG x847 (real bug found, not yet fixed)
- IP-EGG line skipped as "Low Stock" even though Sync Console
  showed exactly 847 in WASP.
- Root cause: `mo-adjust/index.ts:126` — when recipe line has
  `lot=null`, `waspGetAvailableQty` excludes every WASP row that
  has a lot. So lot-tagged stock (auto-assigned by upstream MO)
  doesn't count toward "available".
- Fix: remove that single guard line. `waspRemoveItem` doesn't
  pass a lot to WASP either, so summing across lots gives the
  correct answer.
- Doc: `active/MO/2026-04-24-mo-adjust-no-lot-skips-lot-tagged-stock.md`.
- Same pattern likely exists in `f5-adjust`, `st-adjust`,
  `po-adjust` — flagged for audit.

### SA-2367299 multi-lot collapse (silent inventory drift)
- WASP user removed 14 EGG-X. WASP split it into 5 transactions
  across 5 lots (HID260317/310/303/227/130). Activity log shows
  only 1 entry (qty=2 lot HID260317). The other 4 are missing.
- Root cause: WASP webhook template sends `Lot = "{trans.Lot}"`
  literal placeholder. `lookupWaspTransactionLot` fallback uses
  `(sku, site, location, AssetTransEntryDate, qty)` — but all 5
  transactions share the same `AssetTransEntryDate` and the
  lookup returned `HID260317` for every webhook.
- Net: Katana got −2 from 1 lot; WASP debited −14 across 5.
  Drift = 12 EGG-X across 4 lots, completely silent.
- Doc: `active/SA/2026-04-24-multi-lot-sa-collapses-to-one-lot.md`,
  with 4 fix candidates (WASP template, qty match in fallback,
  drift detector, aggregate-by-saId).
- Manual recovery captured: 4 manual Katana SAs needed to
  re-balance HID260310 / 303 / 227 / 130.

## Decisions and patterns established this session

1. **Daily branches** — `001-WED-...`, `002-THU-...`, `003_FRI_...`
   for snapshot of each work day.
2. **Two-bucket fix folder** — `solved/` for shipped, `active/`
   for in-flight. Each per-flow subfolder mirrors across both.
3. **Per-fix doc format** — Symptom → Evidence (with screenshots
   when possible) → Root cause → Fix (code/SQL) → Verification
   → Manual recovery → Related findings.
4. **Cross-flow root causes**:
   - `katanaGet` swallows non-OK as null. Fixed locally in F5,
     but pattern exists across PO/MO/ST/SA. Worth a centralized
     fix.
   - WASP/Katana state-comparison code over-strict on edge cases
     (sub-unit float, missing lots, etc.). Hysteresis pattern
     applied in PO; needs audit in others.
   - Per-line "Resync" actions (planned/in-progress) will replace
     the auto-retry pattern as the primary recovery path.
5. **Operator workflow** — once Resync is universal across all
   flows, the "adjust from Katana tab in Sync sheet" workaround
   gets removed (committed direction from user).
